# Audit 2026-05-28: Dead code, test coverage, app-flow verification

This report is the output of the Phase 2 audit triggered by the question "any dead code or dead unused ideas? do test cases make sense?" Results are based on live grep + read against the codebase at commit `c1a5d79` (main branch tip) plus actual test-suite runs.

## TL;DR

- **Static health:** good. `make lint` reports 0 issues. `make test` (unit) all pass at 48.4% total coverage; many shared packages at 100%.
- **Integration suite:** 2/13 categories pass cleanly. 10 categories fail with Testcontainers infrastructure issue (Vault container start timeout — gcloud docker-credential-helper interfering, not a code regression). 1 category (`syncerror`) fails with a **genuine test design bug** (INT-SYNC01 tries to bypass CRD validation).
- **E2E suite:** 174/174 ran passed (16 pending intentionally, 8 skipped on env gates). Full reconcile-against-live-Vault loop is green for drift, conflict adoption, cluster variants, discovery, cleanup, cascade, validation rejection, metrics — every documented FLOW.
- **Dead code:** narrow. Only 2 clear cases (`NewNotFoundError`, `NewConnectionError` constructors). Several earlier-suspected items were over-claimed by the static-analysis pass and refuted on re-verification.
- **Dead architectural ideas:** 1 confirmed (event bus has production publishers but no production subscribers — events fly into the void). Already in IMPROVEMENTS §27.
- **IMPROVEMENTS.md resolution status:** §1, §2, §4 are now resolved (per cmd/main.go and feature code); not yet marked.
- **Test quality concerns:** `test/integration/security/input_validation_test.go` uses conditional assertions that pass regardless of webhook state; INT-SYNC01 is a test that cannot exercise its stated scenario.
- **Convention drift:** none observed in TC-ID adherence (100% of e2e tests follow `TC-{CAT}{NN}`).

## Dead code

### Confirmed dead

**`NewNotFoundError` constructor** — `shared/infrastructure/errors/errors.go:141`
Only reference is the definition itself. The corresponding type `NotFoundError` (line 125) and checker `IsNotFoundError` (line 150) **are** used: the checker is called from `shared/controller/syncerror/handler.go:141`. So the type and checker are alive; only the constructor is unused.

**`NewConnectionError` constructor** — `shared/infrastructure/errors/errors.go:175`
Same pattern: type alive, checker called from `features/connection/controller/handler.go:932` and `shared/controller/syncerror/handler.go:146`. Only the constructor is unused.

**Why this matters less than the agent claimed:** the checkers ARE used in production for classifying errors from elsewhere (Vault SDK, K8s API). The custom errors might be intended as a hand-off mechanism that never got wired — or the checkers are matching wrapped errors via `errors.As` (verify before removing). Recommend keeping the types and checkers; consider removing the constructors only after a wider sweep confirms they aren't planned to be used.

## Dead architectural ideas

### Half-wired: event bus (publish without subscribe)

`shared/events/bus.go` exports `Publish` and `PublishAsync`. Both are called in production at 4 sites:
- `features/policy/controller/ops.go:225` — `events.NewPolicyCreated`
- `features/policy/controller/ops.go:236` — `events.NewPolicyDeleted`
- `features/role/controller/ops.go:266` — `events.NewRoleCreated`/`Updated` (varies)
- `features/role/controller/ops.go:277` — `events.NewRoleDeleted`

There are **zero production `Subscribe` calls**. Subscribers exist only in tests. The architectural intent (per FLOW_EVENTS.md and ADR 0004) was inter-feature signaling; in practice nothing consumes the events.

**Status:** matches IMPROVEMENTS §27. Decision needed: either wire production subscribers (e.g., audit logging, cross-feature reconcile triggers, metric tags) or remove the `Publish*` calls. **This is a stakeholder decision** — not a cleanup. Defer to a follow-up plan.

### Conditionally dead (now alive): cleanup queue + orphan metrics

IMPROVEMENTS §31 lists `CleanupQueueSizeGauge`, `CleanupRetriesTotal`, and orphan-related metrics as conditionally dead — alive only if the cleanup and orphan controllers are wired. Spot-check:
- `cmd/main.go:376-383` — cleanup queue + controller wired
- `cmd/main.go:406-415` — orphan detection controller wired

Both are wired today (post the Phase 1 SOLID refactor + cleanup-retry landings). So §31 is **resolved**.

## Test quality concerns

### INT-SYNC01: tests a scenario it cannot trigger

**File:** `test/integration/syncerror/syncerror_test.go:37-66`

The test creates a `VaultPolicy` with `Rules: []vaultv1alpha1.PolicyRule{}` (empty rules array) and asserts that the K8s create succeeds. It then expects the operator's reconciler to set the status to `Error` with `ValidationError`.

But the CRD has `+kubebuilder:validation:MinItems=1` on `Rules`, so the K8s API rejects the create with `spec.rules: Invalid value: 0: spec.rules in body should have at least 1 items` **before the reconciler ever sees the object**.

The test is **structurally broken** — it cannot exercise its stated scenario. To fix:
- Choose an HCL pattern that passes CRD validation but fails Vault validation (e.g., an unsupported capability for a sys path; a path pattern Vault rejects but the CRD allows).
- Or remove this test — the scenario it claims to cover is already enforced at the K8s API layer, so the operator's classifier never has to deal with it.

Tracked as a new IMPROVEMENTS entry in this PR.

### Conditional assertions in `input_validation_test.go`

**File:** `test/integration/security/input_validation_test.go` (multiple sites)

Pattern (line ~38–48):
```go
err := testEnv.K8sClient.Create(ctx, policy)
// The webhook should reject this - path contains invalid characters
if err == nil {
    // If creation succeeded (webhook not running), verify it doesn't bypass
    defer func() { _ = testEnv.K8sClient.Delete(ctx, policy) }()
}
// Note: With webhook enabled, this should fail validation
```

The test passes regardless of webhook state — it makes no `Expect` calls on the observable outcome. If the webhook is disabled, the create succeeds and the test cleans up; if enabled, the create fails and the test does nothing. Either way: green tick, zero signal.

Recommended fix: gate the entire `It(...)` block on webhook availability (skip with reason if absent), then make the assertion unconditional inside.

Tracked as a new IMPROVEMENTS entry in this PR; light-touch fix applied where safe (see Phase 2.4).
